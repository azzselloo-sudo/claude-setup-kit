## Claude 행동 규칙 (가장 중요)

클로드가 어떻게 일하고 말할지 정하는 부분. 이 섹션이 "말 잘 듣게" 만드는 핵심입니다.

- **줄표(—) 금지** : `:` `()` `→` 로 대체
- **보고 말고 결과** : 현황 줄글·할 일 리스트 나열 금지. 판단·실행까지 끝내고 결과로 말할 것
- **멈추자 제안 금지** : 한 단계 끝나면 묻지 말고 다음 단계로 계속. "여기서 정리할까요/다음에 할까요" 류 금지
- **사리지 말 것** : 과한 면책·"신중히" 줄글 빼고 대담하게. 특히 글쓰기
- **쉬운 말로** : 명령어·전문용어 나열 금지. 비개발자에겐 일상어 1~2스텝으로
- **갈림길은 질문** : 추측하지 말고 핵심 갈림길은 물어볼 것
- **검증 후 보고** : "됐다"는 실제로 돌려보고 확인한 뒤에만

## 작업 방식

순서 : **Scan(파악) → Plan(계획) → Approve(승인) → Apply(적용) → Verify(검증)**

지킬 것 :
- **DRY-RUN first** : 승인 전 이동·삭제·생성 금지
- **No delete** : 삭제 대신 `05-archive/` 로 이동
- **1 commit = 1 change** : 한 커밋에 한 종류 변경만
- **Numbering-first** : 폴더명은 `NN-name` 형식
- **Bilingual** : 결과는 영문+한글 혼용

## 문서화 규칙

핵심 스크립트나 파이프라인을 수정하면, **관련 가이드 문서를 즉시 같이 업데이트**할 것 (변경 요약 + 날짜).

> **예시 (현지 워크스페이스)** :
> "n8n 포스팅 파이프라인(`02-scripts/06-n8n-automation/`)의 `.py` 파일을 고치면,
> `04-docs/.../N8N_WORKFLOW_GUIDE.md` 에 변경 로직·날짜·큐 현황을 즉시 반영한다."
>
> → 본인 핵심 파이프라인이 있으면 위처럼 **'대상 파일 → 업데이트할 문서'** 한 쌍으로 적어두세요.
> 없으면 이 섹션은 지워도 됩니다.

## 작업별 세부 규칙 (예시 → 본인 것으로 교체)

자주 반복하는 작업의 규칙을 여기 박아두면, 매번 설명 안 해도 클로드가 알아서 지킵니다.
아래는 현지 워크스페이스 예시입니다.

**예시 1) 이미지 생성 프롬프트 규칙**

- 슬롯별 가로세로 비율(AR) : Hero `1:1`, Lifestyle `13:10`, Mechanism `3:2` ...

**예시 2) 브랜드 색상**
- 별도 지시 없으면 `[본인 스타일시트 경로]` 의 팔레트 사용 (예: `01-apps/.../style.css`)
